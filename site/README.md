# /site/

The rendering pipeline. Everything in here is what GitHub Pages serves — HTML pages, CSS, JS, generated content, and assets. The manuscript itself lives in [`/spine/`](../spine/) and is read-only from this side.

## What's here

| File | Role |
| --- | --- |
| `index.html` | The book's homepage — title, membrane, notochord, spine list, Copy button, WIP badge, footer. |
| `iris.html` | The design-system catalog. Living reference of the colors, type, components, and marks the book wears. |
| `section.html` | Legacy redirect shim — turns old `section.html?n=N` URLs into `section-N.html`. |
| `section-0.html` … `section-18.html` | Pre-rendered per-unit pages. Generated by `build-content.js` from `tools/section-template.html` + the manuscript. |
| `styles.css` | The book's stylesheet. Single source of truth for the design system. |
| `iris.css` | Page-local styles for the iris catalog only (swatch grid, sticky eye, ds-sample wrapper, etc.). |
| `render.js` | Helpers for the index page (membrane disclosure, copy-to-clipboard, etc.). |
| `iris.js` | The sticky-eye behavior on the iris page. |
| `theme-toggle.js` | Light/dark theme toggle in the header. |
| `build-content.js` | The build script. Reads `/spine/*.md`, writes `content.js`, regenerates `section-N.html` from the template, restamps cache-bust hashes. |
| `stamp-assets.js` | Hashes the CSS/JS bundle, rewrites `?v=…` query strings on every `site/*.html`. Idempotent. |
| `content.js` | Generated. The JS data shape (`window.SPINE.{meta, notochord, membrane, tailbone, spine[], atlas}`) the browser pages consume. |
| `build.js` | Generated at deploy time (and locally by `tools/gen-build-js.sh`). Defines `window.SPINE_BUILD = { count, sha }` for the "git #N" label. Gitignored. |
| `assets/` | Web assets — favicon, apple-touch-icon, OpenGraph image, robots.txt. |

## Build flow

1. **Manuscript edit** → user changes a `/spine/*.md` file.
2. **Pre-commit hook** → `tools/hooks/pre-commit` detects the staged markdown and runs `node site/build-content.js`.
3. **build-content.js** reads all manuscript files, writes the fresh `site/content.js`, regenerates the 19 `site/section-N.html` files from `tools/section-template.html`, then invokes `site/stamp-assets.js`.
4. **stamp-assets.js** hashes the asset bundle and rewrites every `?v=…` in `site/*.html` to match.
5. **Pre-commit hook** stages `content.js` and any restamped HTML alongside the original markdown change — one commit, no follow-ups.
6. **Push to main** → `.github/workflows/pages.yml` triggers, writes `site/build.js` with the commit count and SHA, uploads `/site/` as the Pages artifact, deploys.

## Running the build by hand

```
node site/build-content.js
```

Re-runs steps 3–4 above. Safe to run any time; idempotent.

If you need the "git #N" label to match your local HEAD (e.g. running a local preview):

```
tools/gen-build-js.sh
```

That writes `site/build.js` from current git state. Gitignored, never committed.

## Why a separate `/site/` folder

The spine is the load-bearing axis of the book. HTML/CSS/JS that displays it isn't part of the body — it's the surface the body wears. Keeping them separate makes "edit the manuscript" and "edit the rendering" two unambiguous workspaces, and keeps the manuscript directory pure markdown for any future archive / print / migration pass.
